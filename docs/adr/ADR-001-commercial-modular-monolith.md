# ADR-001: 商用化采用模块化单体架构

**状态**: Accepted
**日期**: 2026-04-07
**决策者**: 项目负责人

## 背景

当前系统为可演示 MVP，需要升级为可部署、可审计、可扩展的商用版本。需要在架构风格、数据库、文件存储、任务模型上做出明确决策。

## 决策

### 1. 为什么继续采用模块化单体

**选择**: 保持 FastAPI + Next.js 单体架构，不拆微服务。

**理由**:
- 当前团队规模小（1-2人），微服务的运维和调试开销远大于收益
- 业务域（资产定价、沙盘模拟、驾驶舱）耦合度高，共享数据模型多
- 单体内按模块（api/services/repositories/models）分层，已足够支撑清晰的关注点分离
- 未来如需拆分，模块化单体的仓储层接口可以直接映射为服务边界

**否决方案**: 微服务 — 增加部署复杂度、网络延迟、分布式事务问题，当前阶段不值得。

### 2. 为什么选择 PostgreSQL 而不是继续 SQLite

**选择**: 生产环境迁移到 PostgreSQL，SQLite 仅作为本地开发兼容层。

**理由**:
- SQLite 不支持并发写入，多用户同时操作时会出现锁冲突
- SQLite 无法做细粒度权限控制和行级安全
- PostgreSQL 支持 JSON 字段、全文搜索、LISTEN/NOTIFY，适合未来扩展
- Alembic 迁移体系确保 schema 变更可追踪、可重复

**否决方案**: 继续 SQLite — 不适合多租户、多用户的商用场景。

### 3. 为什么用对象存储替代本地磁盘

**选择**: 引入统一存储抽象层（local/S3），上传文件和报告走对象存储。

**理由**:
- 本地磁盘存储不可在多实例间共享
- 文件丢失无法恢复，无版本控制
- S3 兼容存储（MinIO/阿里云 OSS）提供持久化、CDN、权限控制
- 抽象层允许本地开发仍用磁盘，生产切到 S3，无需改业务代码

**否决方案**: 继续本地磁盘 — 单点故障，不适合生产部署。

### 4. 为什么用 Redis 异步任务而不是同步长请求

**选择**: 资产包计算和报告生成改为异步任务（Redis + arq）。

**理由**:
- 批量估值调用车300 API 可能耗时 10-30 秒，同步请求会超时
- 报告生成调用 LLM 也是长耗时操作
- 异步任务允许前端显示进度、支持重试、不阻塞其他请求
- Redis 同时可用于缓存估值结果，一举两得

**否决方案**: 继续同步 — 用户体验差，容易超时，无法追踪任务状态。
