# Approval Closure Design

## 背景

商业化控制中台第一阶段已经具备：

- 高成本能力守门
- 审批请求数据表与后台审批页
- 成本计量和 usage_event
- 套餐、订阅、预算、触发规则配置

但当前“高级车况定价”在业务侧仍然缺少完整闭环：

- 被拦截时，业务页只能看到通用错误或静默降级
- 操作员需要手工切到后台发起审批
- 审批通过后，业务页面无法直接带审批结果重试
- 审批单与实际执行之间缺少清晰的消费关系

## 目标

先把“高级车况定价”做成完整审批闭环，在不引入复杂工作流引擎的前提下，实现：

1. 业务页被拦截时可直接发起审批
2. 审批通过后可在原业务页重试高成本动作
3. 后端能够校验审批单是否匹配当前租户、动作和对象
4. 审批单被执行后可标记为已消费，避免无限复用
5. 审批结果、降级原因和执行结果可追溯

## 方案对比

### 方案 A：纯手工旁路审批

业务页只提示“去后台发起审批”，后台继续手工填写对象和原因。

优点：

- 改动最小
- 不需要新增审批校验逻辑

缺点：

- 上下文容易丢失
- 业务闭环不成立
- 无法从审批结果回流到原页面

### 方案 B：业务内发起审批 + 审批后原页重试

业务页在高成本动作被拦截时直接拿到结构化审批建议，可一键创建审批单；审批通过后，原页面继续发起带 `approval_request_id` 的重试请求。

优点：

- 业务体验完整
- 复杂度可控
- 可复用现有审批页和 RBAC
- 易于测试

缺点：

- 需要补审批单消费校验
- 前端需要增加一块审批状态 UI

### 方案 C：审批通过后自动恢复执行

被拦截后自动建单，审批通过后自动恢复执行原请求。

优点：

- 操作体验最好

缺点：

- 需要异步任务恢复、幂等控制、状态订阅
- 当前阶段复杂度过高

## 推荐方案

选择方案 B。

这是当前阶段最适合商用落地的平衡点：能真正形成闭环，又不会把系统一下子推进成复杂工作流平台。

## 设计范围

第一阶段仅覆盖高级车况定价相关动作：

- 单车估值 `/api/valuation/single`
- 资产包 AI 建议价 `/api/asset-package/suggest-buyout`
- 资产包计算中触发的高级车况定价

不覆盖：

- 审批通过后自动恢复任务
- 消息通知中心
- 通用工作流编排
- 所有高成本动作的一次性接入

## 后端设计

### 1. 审批上下文结构化返回

当高成本动作被守门拦截，后端不再只给出通用 `HIGH_COST_ACTION_BLOCKED` / `BUDGET_EXCEEDED` / `QUOTA_EXCEEDED`，而是补充标准化 `approval_context`：

- `recommended`: 是否建议发起审批
- `approval_type`
- `reason`
- `related_object_type`
- `related_object_id`
- `estimated_cost`
- `metadata`

### 2. 业务请求支持审批单重试

需要高级车况定价的业务请求新增可选字段：

- `approval_request_id`

当存在 `approval_request_id` 时，后端校验：

- 审批单存在
- 审批单属于当前租户
- 审批单状态为 `approved`
- 审批单尚未被消费
- 审批单类型和对象与当前请求匹配

校验通过后，本次请求允许执行高级车况定价。

### 3. 审批单消费

审批单一旦被成功用于高成本执行，需要记录：

- `consumed_at`
- `consumed_request_id`

这样能明确“一张审批单已经被用于哪次请求执行”，避免无限复用。

### 4. usage_event 和审计

高成本执行成功时，把以下内容写入 `usage_event.metadata`：

- `approval_request_id`
- `approval_consumed`
- `requested_level`
- `executed_level`
- `degraded`
- `reason`

审批创建、审批通过、审批拒绝、审批消费继续走审计日志。

## 前端设计

### 1. 业务页审批卡片

资产包页在高成本建议/执行被拦截时展示审批卡片，包括：

- 当前为什么被拦截
- 预计成本
- 当前是已降级还是完全阻断
- 发起审批按钮
- 当前审批状态
- 审批通过后的“重试高级车况定价”按钮

### 2. 资产包页行为

重点接入资产包页：

- `AI 建议价` 被高成本策略拦截时，可直接发起审批
- 审批通过后可用 `approval_request_id` 重试获取 AI 建议

页面上保留“继续使用基础估值/普通流程”的路径，不强制卡死。

### 3. 审批后台页增强

审批页增加更明确的业务信息展示：

- 关联对象
- metadata 摘要
- 是否已消费
- 消费时间

## 测试策略

至少覆盖：

- 被预算/额度/规则拦截时返回结构化审批上下文
- 带 `approval_request_id` 且审批通过时允许高级车况定价执行
- 已消费审批单不能重复执行
- 资产包页能从被拦截状态发起审批并识别审批通过后的可重试状态

## 风险与约束

- 这轮不做自动恢复执行，所以仍然需要操作员主动点一次“重试”
- 如果未来扩到更多高成本动作，审批单类型和对象约束需要继续标准化
- 现有 `approval_requests` 结构比较轻，本轮只做最小可用扩展，不引入复杂状态机
